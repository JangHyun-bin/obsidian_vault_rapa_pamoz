영역 A

**사전 확인 필요 사항**

회의 본격 진행 전, 레몬헬스케어 측에 직접 질의하여 확인이 필요한 정보들입니다. 아래 항목이 명확해져야 이후의 아키텍처 논의가 의미를 갖습니다.

A — 1

**레몬케어 앱의 기술 스택 확인**

배경

파모즈의 측위 · AI SDK 형태는 레몬케어 앱이 채택한 기술 스택에 따라 결정됩니다.

확인 사항

* 순수 네이티브(Swift + Kotlin) 구성 여부
* 크로스플랫폼 프레임워크(React Native / Flutter / Kotlin Multiplatform) 사용 여부
* 현재 앱의 최소 지원 OS 버전

결정 영향

네이티브인 경우 파모즈는 .xcframework(iOS) + .aar(Android) 형태로 SDK 제공. 크로스플랫폼인 경우 TurboModule, FFI, 또는 플러그인 형태로 별도 패키징 필요.

우선순위 **최우선** 본 항목 미확정 시 SDK 설계 착수 불가

A — 2

**병원 내부망 'AI 시스템' 박스의 실체 정의**

배경

최근 공유된 시스템 구성도 우측 하단의 'AI 시스템' 블록이 실제로 어떤 컴포넌트로 구성되는지 명확하지 않습니다.

확인 사항

* 병원 내부 sLLM(8B~13B급 추론 서버)을 의미하는지
* 단순 RAG 매니저 / 오케스트레이션 레이어인지
* 외부 LLM 호출 wrapper 역할만 하는지

결정 영향

PHI(개인의료정보) 포함 쿼리의 LLM 추론이 병원 내부에서 일어나는지, 클라우드로 송신되는지를 결정합니다. 의료법 준수 여부와 직결된 사안입니다.

우선순위 **최우선**

A — 3

**QAB(레몬케어 API 서버)의 정의와 API 스펙 공유**

배경

QAB 서버가 병원 내부망에서 EMR/OCS/PACS 및 편의시스템과 연결된 통합 API 게이트웨이로 보이나, 정확한 역할 정의와 노출 API 명세가 공유되지 않은 상태입니다.

확인 사항

* QAB가 제공하는 API의 종류(EMR 조회, 처방 조회, 순번/수납 조회 등)
* 인증 방식 및 호출 권한 모델
* 외부 시스템(파모즈 RAG/AI 서버)의 QAB 호출 가능 여부 및 정책

결정 영향

파모즈의 RAG 시스템이 QAB 위에 얹히는 형태로 가면 EMR/PACS 직접 연동이 불필요해져 보안 · 개발 부담이 크게 감소합니다.

우선순위 **고**

A — 4

**측위 시스템 개발 주체 및 협력사 식별**

배경

현재 시스템 구성도에 '측위 관련 서버'가 명시되어 있으나, 해당 서버의 개발 · 운영 주체가 명확하지 않습니다.

확인 사항

* 측위 시스템을 레몬헬스케어가 자체 개발하는지
* 외부 측위 솔루션 협력사가 별도 참여하는지
* 파모즈가 측위 풀스택(SDK + 백엔드 + 모델)을 책임지는 구조인지

결정 영향

본 항목이 정해져야 인터페이스 계약 작성, 책임 분담, 비용 산정이 가능합니다.

우선순위 **최우선**

영역 B

**아키텍처 정렬**

레몬헬스케어 측 제안 시스템 구성도와 파모즈가 검토한 측위 · RAG · LLM 통합 구조 사이의 주요 차이점을 다룹니다. 본 영역의 합의가 이후 개발 일정 · 비용 산정의 기준선이 됩니다.

B — 1

**클라우드 GPU 전략 (1안/2안 방향성 재검토)**

레몬헬스케어 안

1안(본 과제 수행기) — 각 병원 내부에 고성능 GPU 및 RAG DB 온프레미스 배치

2안(향후 확장기) — 클라우드 중앙 GPU LLM 또는 외부 API(예: Google API) 활용

파모즈 검토 의견

일반적인 의료 AI 배포 패턴 대비 방향성이 역순으로 판단됩니다.

특히 2안의 '구글 API' 옵션은 **의료법 · 개인정보보호법상 사실상 적용 불가**합니다. 환자 의료정보의 국외 이전 동의 확보 자체가 어렵습니다.

PHI 포함 쿼리는 LLM 추론 시 반드시 의료기관 통제 영역 내에서 처리되어야 합니다.

대안 제안

1안(본 과제) — 네이버 클라우드 내 공동 GPU 풀 구축, 다수 병원이 공유(비용 효율적)

2안(확장기) — 병원별 온프레미스 GPU 분산 배포(지연 · 데이터 주권 개선)

결정 필요

1안/2안 방향성 합의, 외부 API 활용 가능 여부에 대한 법무 자문 결과 반영.

B — 2

**AI 챗봇 호출과 측위 호출의 채널 분리**

배경

현재 시스템 구성도는 측위 호출과 AI 챗봇 호출의 흐름이 단일 직렬 경로로 표현되어 있습니다.

파모즈 검토 의견

* 측위 데이터는 1–5Hz의 고빈도 실시간 업데이트가 요구되며, 매 호출마다 LLM 및 RAG를 거치는 경로는 부적절
* 측위 사용 시나리오 중 다수는 LLM 호출이 불필요한 단순 위치 조회로 종료됨
* AI 챗봇 호출과 측위 호출은 서로 다른 빈도 · 지연 요구 · 데이터 특성을 가지므로 채널 분리 필요

요청 사항

레몬헬스케어 측에 다음 두 가지 흐름도의 별도 작성을 요청:

* **AI 챗봇 호출 시퀀스** — 위치 컨텍스트 활용 / 비활용 케이스 분기 포함
* **측위 호출 시퀀스** — 환자 앱 → 병원 측위 백엔드 직통 경로

결정 필요

흐름도 분리 작성 합의 및 측위 채널의 클라우드 우회 여부 정책 합의.

B — 3

**측위 백엔드의 배치 위치 (클라우드 vs 병원)**

레몬헬스케어 안

측위 관련 서버를 네이버 클라우드 측에 배치하는 구조.

파모즈 검토 의견

* 측위 인프라(WiFi AP, BLE 비콘, 핑거프린팅 맵)는 병원 건물 자산이며 본질적으로 병원 종속적
* 환자 위치 데이터는 의료기록과 결합 시 PHI에 준하는 민감 정보로 분류 가능
* 1–5Hz 실시간 처리를 위해 클라우드 라운드트립 지연을 최소화할 필요
* 따라서 측위 백엔드의 병원 온프레미스(또는 DMZ) 배치 권장

측위 백엔드 권장 구성

* 센서 스트림 수신 게이트웨이 — WiFi/BLE/IMU 스트림, 윈도우 버퍼링
* DL 측위 추론 서버 — Res-Transformer-LSTM 등, GPU 기반
* 융합 · 필터 엔진 — EKF/Particle Filter, IMU dead reckoning, QR 앵커 보정
* 측위 DB — 핑거프린팅, 비콘 좌표, 실내지도, POI
* 위치 이벤트 Pub/Sub — MQTT 또는 WebSocket

결정 필요

측위 백엔드의 배치 위치 합의, 측위용 GPU 자원 산정(현재 구성도에 누락된 항목).

B — 4

**PHI 라우팅 정책 및 가드레일 배치**

배경

환자 질의의 성격에 따라 LLM 추론 위치 및 가드레일 통과 경로가 분기되어야 합니다.

라우팅 제안

* **Non-PHI 일반 안내** (의료 상식, 일반 안내) → 클라우드 중앙 LLM 처리
* **PHI-touching 질의** (환자별 처방, EMR 조회 등) → 병원 내부 sLLM 처리
* 라우팅 결정은 AI 기능 서버(클라우드)에서 분기

가드레일 2단 구조 제안

* **1단 — 클라우드 입력 가드레일** — PII 1차 필터, 악성 프롬프트 차단
* **2단 — 병원 입력 가드레일** — PHI 검증, 의료 안전성 추가 검증
* **출력 가드레일** — 병원 내부 위치(현행과 동일)

결정 필요

PHI 분기 기준 정의, sLLM 도입 여부 및 모델 선정, 가드레일 단계 합의.

영역 C

**모바일 통합**

기존 레몬케어 앱에 파모즈의 측위 · AI 기능을 통합하기 위한 SDK 형태 · 책임 분담 · 기술 제약 사항을 다룹니다.

C — 1

**iOS WiFi 스캔 제약의 컨소시엄 공식 통보**

기술적 사실

iOS 운영체제에서는 일반 앱이 **주변 WiFi AP 스캔(SSID/BSSID/RSSI 수집)을 수행할 수 없습니다.**

* NEHotspotHelper API는 Apple로부터 별도 특수 entitlement 승인이 필요하며, 의료 앱이 승인받기는 사실상 불가능
* 현재 연결된 WiFi의 SSID 정도만 제한적으로 조회 가능
* Android는 12+ 버전에서 NEARBY\_WIFI\_DEVICES 권한 부여 시 정상적으로 WiFi 스캔 가능

영향

* iOS에서는 WiFi 핑거프린팅 기반 측위가 불가능
* iOS 측위는 BLE 비콘 + IMU + QR 앵커 중심 알고리즘으로 설계되어야 함
* DL 측위 모델의 입력 차원이 iOS / Android 간 상이
* iOS 측위 정확도는 구조적으로 Android 대비 낮을 수 있으며, 비콘 배치 밀도를 iOS 기준으로 산정 필요

요청 사항

본 제약 사항을 컨소시엄 차원에서 공식 인지하고, 측위 정확도 KPI 산정 단계에서 OS별 별도 기준을 반영해야 합니다.

우선순위 **고** 알고리즘 설계 및 KPI 정의에 즉시 반영 필요

C — 2

**SDK 통합 모델 및 책임 분담**

통합 모델 제안

* **앱 owner — 레몬헬스케어** (메인 앱 코드베이스, 인증/결제/푸시 인프라)
* **SDK provider — 파모즈** (측위 · AI 챗봇 SDK 및 백엔드)
* **통합 방식 —** .xcframework(iOS) + .aar(Android) 형태로 SDK 제공, 레몬헬스케어가 자사 앱에 의존성 추가

빌드 · 사이닝

* iOS — Apple Developer Program 계정은 레몬헬스케어가 보유한 것 그대로 사용. 파모즈는 사이닝 미관여
* iOS — Privacy Manifest(iOS 17+ 필수), 권한 키(블루투스/위치/모션/카메라) 명세 사전 전달
* Android — AAR 배포. AndroidManifest 권한 및 ProGuard rules 명세서 사전 전달

배포 채널

사내 Maven 저장소 또는 GitHub Packages를 통해 SDK 버전 관리. 시맨틱 버저닝 엄격히 적용(Breaking change 시 메이저 버전 증가).

결정 필요

SDK 통합 모델 합의, 배포 채널 선정, 권한 키 사전 협의.

C — 3

**iOS / Android별 네이티브 SDK 별도 개발**

배경

센서 접근 API, 권한 모델, 백그라운드 동작 정책이 OS마다 근본적으로 다르며, 측위 입력 채널 자체가 OS별로 상이합니다.

OS별 분리 필수 영역

* 센서 수집 모듈(BLE / IMU / WiFi / 카메라)
* 권한 요청 플로우 및 백그라운드 동작 처리
* 온디바이스 추론 런타임 — iOS는 Core ML, Android는 TFLite 또는 ONNX Runtime
* 챗봇 UI 컴포넌트 — SwiftUI / Jetpack Compose

공유 가능 영역

* 백엔드 API 스펙 — OpenAPI 단일 명세, 양쪽 SDK 자동 생성
* 모델 아키텍처 — Res-Transformer-LSTM 동일, 가중치는 변환만 별도
* 학습 파이프라인 및 학습 데이터
* 데이터 스키마 — 센서 페이로드, 위치 응답 포맷

인력 요건

iOS 개발자 및 Android 개발자 각 1명 이상 별도 산정 필요. 측위 알고리즘 개발자와 모바일 개발 인력은 분리하여 운영.

결정 필요

컨소시엄 차원의 모바일 개발 인력 산정, OS별 KPI 기준 별도 수립.

영역 D

**개발 협업 체계**

기관 간 효율적인 병렬 개발과 안정적인 통합을 위해 사전에 합의해야 할 협업 프로세스 및 인프라 사항을 다룹니다.

D — 1

**인터페이스 계약 우선 작성**

원칙

코드 작성 이전에 SDK API 시그니처와 백엔드 OpenAPI 스펙을 먼저 합의하여, 양 기관이 mock 기반으로 병렬 개발이 가능하도록 합니다.

작성 대상

* SDK 공개 API 시그니처 — Swift 및 Kotlin
* 측위 백엔드 OpenAPI 3.0 명세 — 센서 스트림 ingest, 위치 조회, 이벤트 구독
* AI 챗봇 백엔드 OpenAPI 3.0 명세 — 질의 송신, 스트리밍 응답, 세션 관리
* 데이터 스키마 — JSON Schema 형태로 별도 명세

권장 도구

OpenAPI Generator를 통해 양쪽 SDK 클라이언트 코드 자동 생성. Postman/Bruno collection으로 mock 서버 공유.

결정 필요

인터페이스 계약 초안 작성 일정, 검토 · 합의 프로세스.

D — 2

**백엔드 통합 포인트 합의**

통합 포인트

**파모즈 SDK** ↔ **레몬케어 인증서버** OAuth2 SSO 토큰

**AI Proactive 안내** ↔ **레몬케어 푸시서버** 푸시 채널 호출 API

**RAG → EMR 조회** ↔ **QAB API 서버** QAB 경유(EMR 직접 미연동)

**운영 로그** ↔ **레몬케어 로그/모니터링** 로그 수집 endpoint 통합

핵심 권장 사항

파모즈의 RAG 시스템이 EMR/OCS/PACS를 직접 조회하지 않고 **QAB API를 경유하는 구조**가 권장됩니다. 보안 경계가 명확해지고 파모즈 측 개발 부담이 감소합니다.

결정 필요

각 통합 포인트에 대한 인증/권한/SLA 합의.

D — 3

**레포지토리 · CI/CD · 이슈 트래킹**

레포지토리 구조

* 레몬케어 앱 코드 — 레몬헬스케어 소유 레포(파모즈 미접근)
* 파모즈 SDK 코드 — 파모즈 소유 레포(레몬헬스케어 미접근)
* 파모즈 백엔드 코드 — 파모즈 소유 레포(레몬헬스케어 미접근)
* 공유 채널 — SDK 배포 채널(Nexus / GitHub Packages)

CI/CD

* 파모즈 SDK — 자체 CI에서 빌드 → 버전 태그 → 배포 채널 푸시
* 레몬케어 앱 — 자사 CI에서 파모즈 SDK 의존성을 끌어와 통합 빌드
* 환경 분리 — dev / staging / prod 별도 백엔드 endpoint, SDK 빌드 variant로 스위칭

이슈 트래킹

통합 작업용 공유 Jira 또는 Linear 프로젝트 운영. 각 기관 내부 이슈는 별도 관리.

결정 필요

공유 이슈 트래커 선정, CI/CD 파이프라인 인터페이스 합의.

영역 E

**규제 및 리스크**

의료 데이터를 다루는 본 프로젝트의 특성상 사전 검토가 필수적인 법적 · 정책적 리스크 요인을 정리합니다. 본 영역은 별도 법무 자문 트랙으로 진행할 것을 권장합니다.

E — 1

**의료 데이터의 국외 이전 가능 여부 검토**

배경

현 시스템 구성도 2안에 'Google API' 옵션이 명시되어 있으며, 외부 LLM API 활용 시 환자 데이터의 국외 이전이 필연적으로 발생합니다.

관련 법규

* 개인정보보호법 제28조의8(개인정보의 국외 이전)
* 의료법 및 의료기관 개인정보보호 가이드라인
* 보건의료 데이터 활용 가이드라인(보건복지부)

주요 쟁점

* 환자 의료정보의 국외 이전 동의 확보 절차
* 국외 클라우드 사업자의 보안 인증(ISO 27001, HIPAA 등) 충분성
* 의료 데이터 처리 위탁 시의 책임 관계

요청 사항

컨소시엄 차원의 법무 자문 의뢰(전남대학교 산학협력단 또는 외부 법무법인 활용 검토).

E — 2

**측위 데이터의 PHI 분류 여부 명확화**

쟁점

환자 위치 정보 자체는 단독으로는 PHI가 아니나, 환자 식별자 또는 의료기록과 결합되는 순간 민감 정보로 전환됩니다.

검토 사항

* 측위 raw 데이터(WiFi/BLE 스캔, IMU)의 분류 기준
* 환자 ID와 결합된 위치 데이터의 보관 · 활용 정책
* DL 측위 모델 학습용 데이터의 익명화 절차
* 학습 파이프라인이 클라우드에 위치할 경우 익명화 데이터의 이전 가능 여부

권장 정책

환자 ID와 위치 raw 데이터를 분리 보관. 학습용 데이터는 익명화 후에만 외부(클라우드) 송신 허용.

결정 필요

데이터 분류 정책 및 익명화 절차 명문화.

E — 3

**iOS App Store / Google Play 심사 리스크 사전 대응**

주요 리스크

* 위치 · 블루투스 권한이 의료 목적이라는 점의 명시적 정당화 필요
* iOS 17+ Privacy Manifest 누락 시 reject 위험
* Google Play의 위치 권한 정책(Background Location) 심사 강화
* HealthKit / Health Connect 사용 시 추가 심사 절차 발생

사전 준비

* 권한 사용 사유(Usage Description) 문구의 사전 검토
* Privacy Manifest 작성 — 파모즈 SDK 측 manifest 포함
* 의료 앱 카테고리 등록 시 필요 서류 사전 확보
* 베타 배포 채널(TestFlight, Internal Testing)을 통한 사전 심사 리허설

결정 필요

심사 대응 책임 분담(앱 owner인 레몬헬스케어 주관 권장).

마무리

**권장 진행 순서**

위 5개 영역 12개 아젠다는 긴급도와 의존 관계에 따라 다음 순서로 진행할 것을 권장합니다.

즉시 · 회의 전
